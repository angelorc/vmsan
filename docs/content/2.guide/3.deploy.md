vmsan provides a complete deployment workflow for multi-service applications defined in vmsan.toml. A single command resolves dependencies, creates VMs, uploads code, builds, and starts your services.

## Deploy everything

Use `vmsan up` to deploy all services and accessories defined in your project:

```bash
vmsan up
```

This command parses vmsan.toml, resolves the dependency graph using topological sort, and deploys services in parallel within each dependency group. Health checks are verified between groups before proceeding.

## Re-deploy a service

Use `vmsan deploy` to re-deploy application code without recreating VMs:

```bash
# Re-deploy all services
vmsan deploy

# Re-deploy a single service
vmsan deploy web
```

This uploads your source files, runs the build command, and restarts the service. Deploy hash tracking ensures idempotent re-deploys — if the source hasn't changed, the deploy is skipped.

## Stop services

Use `vmsan down` to stop all project services:

```bash
# Stop all services
vmsan down

# Stop and destroy VMs and data
vmsan down --destroy

# Skip confirmation prompt
vmsan down --force
```

::caution
The `--destroy` flag removes VMs and their data permanently. This cannot be undone.
::

## Check status

View a project overview with service status:

```bash
vmsan status
```

This displays a colored table showing each service's current state, health, and resource usage.

## Deployment phases

Each service goes through these phases during `vmsan up`:

| Phase | Description |
| --- | --- |
| Creating | VM creation via gateway |
| Uploading | Source files transferred to agent |
| Building | Build command executed |
| Starting | Start command launched |
| HealthCheck | Service health verified |
| Running | Service is live and healthy |

::note
Accessories (databases, caches) are always deployed first since they sit at the bottom of the dependency graph. Application services wait for their dependencies to pass health checks before starting.
::

## Built-in accessories

vmsan includes pre-built rootfs images for common infrastructure services:

| Accessory | Description |
| --- | --- |
| Postgres 16 | Pre-built rootfs with auto-configured credentials |
| Redis 7 | Pre-built rootfs, ready to use |

Accessories are declared in the `[accessories]` section of vmsan.toml and deployed automatically as part of the dependency graph.

## Example workflow

A typical deployment workflow from start to finish:

```bash
# Generate project configuration
vmsan init

# Validate the configuration
vmsan validate

# Set encrypted secrets
vmsan secrets set DB_PASSWORD=mysecret

# Deploy all services and accessories
vmsan up

# Check service status
vmsan status

# Re-deploy only the web service after a code change
vmsan deploy web

# Stop everything when done
vmsan down
```

::tip
Use `vmsan deploy <service>` after code changes for fast re-deploys. It skips VM creation and only uploads, builds, and restarts the service.
::
