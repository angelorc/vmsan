vmsan supports multi-host deployments via a control plane server and agent-host workers. Deploy VMs across multiple bare-metal servers from a single CLI.

## Architecture

| Component | Role |
| --- | --- |
| **vmsan-server** | Control plane HTTP server (port 6443). SQLite backend. Manages hosts, VMs, and join tokens. |
| **vmsan-agent-host** | Worker process on each host. Joins the control plane, sends heartbeats every 30s, and pulls commands via the sync engine (10s poll). |

## Setup

### 1. Start the control plane

Run the control plane server on your primary host:

```bash
vmsan server
```

The server listens on port 6443 by default and uses SQLite for state storage.

### 2. Add a worker host

Generate a join token and register a new host:

```bash
vmsan hosts add --server http://server:6443 --token <generated-token> --name worker-1
```

::note
Join tokens are single-use and expire after 24 hours.
::

### 3. Start the agent-host

On each worker machine, start the agent-host process:

```bash
vmsan agent-host
```

The agent-host connects to the control plane, sends periodic heartbeats, and executes commands received via the sync engine.

## Managing hosts

### List registered hosts

```bash
vmsan hosts list
```

### Check host health

```bash
# Check all hosts
vmsan hosts check

# Check a specific host
vmsan hosts check worker-1
```

### Remove a host

```bash
vmsan hosts remove <hostId> --server http://server:6443
```

::caution
Removing a host does not automatically stop or migrate VMs running on it. Stop or migrate VMs before removing the host.
::

## Remote VM creation

Create VMs on a specific host via the control plane:

```bash
vmsan create --host server:6443 --runtime node24 --memory 512
```

The control plane schedules the VM on an available worker based on resource availability.

## Sync engine

The sync engine uses a pull-based model for command distribution:

- The agent-host polls the control plane every 10 seconds
- Exponential backoff is applied on connection failures
- Commands are queued on the server and executed on the local gateway when pulled
- Command execution results are reported back to the control plane

::tip
The pull-based model means agent-hosts do not need inbound ports open. They only make outbound HTTP requests to the control plane.
::
