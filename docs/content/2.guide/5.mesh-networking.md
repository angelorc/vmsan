vmsan mesh networking enables VMs within a project to discover and communicate with each other using service names, without exposing ports to the host. Services connect over a private mesh subnet with zero-trust access control.

## Overview

Each VM in the mesh gets a deterministic IP from the `10.201.0.0/16` subnet based on its slot index. Services register a name and declare which other services they need to reach. Only declared connections are permitted.

## Registering a service

Assign a service name when creating a VM:

```bash
vmsan create --runtime node24 --service web
```

Or in vmsan.toml:

```toml
[services.web]
runtime = "node24"
service = "web"
```

## Declaring connections

Specify which services a VM needs to connect to:

```bash
vmsan create --runtime node24 --service web --connect-to "api:3000,db:5432"
```

Or in vmsan.toml:

```toml
[services.web]
runtime = "node24"
service = "web"
connect_to = ["api:3000"]

[services.api]
runtime = "node24"
service = "api"
connect_to = ["db:5432"]

[accessories.db]
type = "postgres"
```

## DNS service discovery

VMs resolve each other by service name using the `.vmsan` domain. The mesh DNS handler runs on the host and intercepts DNS queries from VMs.

Inside the `web` VM, you can reach the `api` service:

```bash
curl http://api.vmsan:3000/health
```

The DNS query `api.vmsan` resolves to the mesh IP assigned to the `api` service.

## L3 routing

IP forwarding between VM network namespaces is handled via veth pairs. Each VM gets a mesh IP in addition to its host IP, enabling direct L3 connectivity within the mesh.

## nftables ACLs

Per-VM allow/deny rules control which services can communicate. Only connections declared via `connect_to` are permitted in the nftables rules.

```bash
# View mesh connections for a project
vmsan network connections
```

## Traffic counters

nftables byte and packet counters are maintained per mesh connection. Use `vmsan network connections` to view traffic statistics between services.

## Security

The mesh is zero-trust by default:

- **Only declared connections are allowed.** If service A does not list service B in `connect_to`, traffic is blocked at the nftables layer.
- **Cross-project mesh traffic is blocked.** VMs in different projects cannot communicate over the mesh.
- **ECH stripping** prevents Encrypted ClientHello from hiding the SNI, ensuring TLS filtering cannot be bypassed.
- **DNS rebinding protection** blocks private IP resolution in DNS responses, preventing VMs from using DNS to reach internal hosts.

::warning
If you forget to declare a connection in `connect_to`, traffic between those services will be silently dropped. Check `vmsan network connections` to verify your mesh topology.
::
