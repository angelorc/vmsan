## What changed

vmsan 0.4.0 adds defense-in-depth egress filtering with three independent layers: DNS filtering, SNI filtering, and L3/L4 nftables. This replaces the DNS-based best-effort domain filtering from previous versions.

### Defense-in-depth egress filtering

Each VM now gets a dedicated dnsproxy sidecar (~15-25 MB RSS) and an in-process tcpproxy, both managed by a new vmsan-gateway daemon. Outbound traffic passes through:

1. **DNS layer** — dnsproxy filters queries, returns NXDOMAIN for non-allowed domains
2. **SNI layer** — tcpproxy inspects TLS ClientHello, blocks non-matching SNI
3. **L3/L4 layer** — nftables enforces IP and port rules

### vmsan-gateway daemon

A new Go binary (`vmsan-gateway`) runs on the host and manages the lifecycle of per-VM dnsproxy and tcpproxy instances. It is started automatically when a VM is created and stopped when the last VM is removed.

### `--allow-icmp` flag

ICMP traffic can now be explicitly enabled per VM:

```bash
vmsan create --allow-icmp
```

Previously, ICMP was always blocked with no override.

### `vmsan logs --dns`

A new command provides visibility into DNS queries made by a VM:

```bash
vmsan logs --dns <vm-id>
```

### Legacy iptables codepath removed

The `VMSAN_LEGACY_IPTABLES=1` environment variable and the legacy iptables fallback have been removed. nftables is now the only supported firewall backend.

### Port range overlap fixed

The HTTP proxy port range has been moved from `10080-10334` to `10698-10952` to eliminate overlap with the DNS proxy range.

## Migration steps

### 1. Stop all running VMs

```bash
sudo vmsan list --json | jq -r '.[] | select(.status == "running") | .id' | xargs -I{} sudo vmsan stop {}
```

### 2. Upgrade vmsan to 0.4.0

```bash
curl -fsSL https://vmsan.dev/install | bash
```

This installs the updated vmsan CLI, vmsan-gateway binary, and downloads dnsproxy to `~/.vmsan/bin/`.

### 3. Start VMs

```bash
sudo vmsan start <vmId>
```

The gateway daemon starts automatically and creates per-VM proxy instances.

### 4. Verify DNS filtering

```bash
vmsan logs --dns <vmId>
```

## Breaking changes

### Legacy iptables removed

The `VMSAN_LEGACY_IPTABLES=1` escape hatch no longer works. If you were using it, you must ensure nftables is available on your host (kernel 5.10+). Run `sudo vmsan doctor` to verify.

### DNS behavior change in `allow-all` mode

In previous versions, DNS queries in `allow-all` mode were forwarded directly to upstream resolvers via simple DNAT. In 0.4.0, they pass through the per-VM dnsproxy. Behavior is identical (all queries are forwarded), but queries are now logged and DNS rebinding protection is active.

### Additional memory per VM

Each VM's dnsproxy sidecar uses approximately 15-25 MB of RSS on the host. Factor this into capacity planning if running many concurrent VMs.

### Port range change

The HTTP proxy reserved port range moved from `10080-10334` to `10698-10952`. If you had any scripts or firewall rules referencing the old range, update them.

### `--allowed-domain` enforcement

Domain filtering is now enforced at three layers instead of one. This means:

- Domains must pass DNS filtering (dnsproxy)
- TLS connections must pass SNI filtering (tcpproxy)
- IP connections must pass nftables rules

In practice, this makes domain filtering much more robust. Direct-IP access to non-allowed hosts is no longer possible when using `custom` policy.
