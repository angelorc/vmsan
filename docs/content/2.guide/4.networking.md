Each vmsan microVM gets its own TAP network device with a `/30` subnet (`198.19.{slot}.0/30`). You can control outbound network access with policies.

::note
Since 0.2.0, vmsan uses nftables with atomic rule application for network isolation. Each VM gets its own nftables table (`vmsan_<vmId>`) with FORWARD, prerouting, and postrouting chains.
::

## Set a network policy at creation

Use the `--network-policy` flag when creating a VM:

```bash
# Allow all outbound traffic (default — observation mode)
vmsan create --network-policy allow-all

# Block all outbound traffic
vmsan create --network-policy deny-all

# Custom policy with allowed domains
vmsan create --network-policy custom --allowed-domain "*.github.com,npmjs.org"
```

### Available policies

| Policy      | Behavior |
| ----------- | -------- |
| `allow-all` | All traffic permitted. DNS and SNI are logged but not filtered. Use for observation and debugging. |
| `custom`    | Only traffic to `--allowed-domain` destinations is permitted. DNS, SNI, and L3/L4 layers all enforce the allowlist. |
| `deny-all`  | No outbound traffic. DNS proxy is not started. |

::note
The policy is automatically promoted to `custom` when you specify `--allowed-domain`, `--allowed-cidr`, or `--denied-cidr`.
::

## Defense-in-depth egress filtering

Since 0.4.0, vmsan applies three independent filtering layers to outbound traffic. A request must pass all three layers to reach the internet.

### Layer 1: DNS filtering (dnsproxy)

Each VM gets a dedicated dnsproxy sidecar process managed by vmsan-gateway. All DNS queries from the VM are DNAT'd to this per-VM proxy on the host (port 53 → host port `10053+slot`).

- In `custom` mode, only queries for allowed domains receive real answers. All other queries return `NXDOMAIN`.
- In `allow-all` mode, all queries are forwarded and logged.
- DNS rebinding protection blocks private IP addresses (RFC 1918, link-local, loopback) in DNS responses, preventing VMs from using DNS to reach internal hosts.

### Layer 2: SNI filtering (tcpproxy)

TLS connections on port 443 are DNAT'd to an in-process tcpproxy on the host (port `10443+slot`). The proxy inspects the TLS ClientHello to extract the SNI (Server Name Indication).

- In `custom` mode, only connections with an SNI matching an allowed domain are forwarded. Connections without SNI or with a non-matching SNI are refused.
- In `allow-all` mode, all TLS connections are forwarded and the SNI is logged.
- ECH (Encrypted ClientHello) SvcParam records are stripped from DNS responses to prevent SNI encryption bypass.

### Layer 3: L3/L4 nftables

The nftables layer enforces IP-level and port-level rules:

- `--allowed-cidr` and `--denied-cidr` control which IP ranges are reachable.
- Cross-VM traffic is always blocked.
- UDP is blocked by default (except DNS).
- ICMP is blocked by default (use `--allow-icmp` to enable).

### How `--allowed-domain` flows through the layers

When you create a VM with `--allowed-domain "*.github.com,registry.npmjs.org"`:

1. **DNS**: dnsproxy only resolves `*.github.com` and `registry.npmjs.org`. All other queries return `NXDOMAIN`.
2. **SNI**: tcpproxy only forwards TLS connections whose SNI matches `*.github.com` or `registry.npmjs.org`.
3. **nftables**: IP-level rules permit traffic to resolved IPs. Direct-IP access to non-allowed hosts is blocked.

## Allow ICMP traffic

ICMP is blocked by default to prevent ICMP tunneling. To allow ICMP (e.g., for `ping`):

```bash
vmsan create --allow-icmp
```

ICMP type 3 (destination unreachable / fragmentation needed) is always accepted via `ct state related` for established connections, preserving Path MTU Discovery (PMTUD).

## Allow specific domains

Restrict outbound traffic to specific domains:

```bash
vmsan create --allowed-domain "api.example.com,*.github.com"
```

Wildcard `*` matches subdomains. Domain filtering is enforced at all three layers (DNS, SNI, nftables).

## Allow or deny CIDR ranges

Control access by IP address ranges:

```bash
# Allow specific ranges
vmsan create --allowed-cidr "10.0.0.0/8,192.168.1.0/24"

# Deny specific ranges (takes precedence over all allows)
vmsan create --denied-cidr "10.0.0.0/8"
```

## Forward ports to a VM

Expose VM ports to the host:

```bash
vmsan create --publish-port 8080,3000
```

This sets up DNAT rules to forward traffic from the host ports to the VM.

## Limit bandwidth

Cap the outbound bandwidth per VM:

```bash
vmsan create --bandwidth 50mbit
```

## Update a policy on a running VM

Change the network policy without restarting:

```bash
vmsan network update <vm-id> --policy deny-all
vmsan network update <vm-id> --policy custom --domains "api.example.com" --allow-icmp
```

The `network update` subcommand accepts `--policy`, `--domains`, `--cidrs`, and `--allow-icmp` flags.

## View DNS logs

Inspect DNS queries made by a VM:

```bash
vmsan logs --dns <vm-id>
```

This shows all DNS queries processed by the VM's dnsproxy sidecar, including the query name, response code, and resolved IPs.

## Security defaults

### ICMP blocked

All ICMP traffic from VMs is blocked by default. This prevents ICMP tunneling. Use `--allow-icmp` to re-enable. ICMP type 3 (destination unreachable) is always accepted via `ct state related`.

### UDP blocked (except DNS)

All UDP traffic is blocked by default, except for DNS (port 53 via DNAT to dnsproxy). This prevents UDP-based data exfiltration. Applications that rely on UDP (QUIC/HTTP3, game servers, streaming protocols) will not work from inside VMs.

### DNS bypass prevention

To prevent VMs from bypassing DNS filtering:

- **All DNS** (port 53) is DNAT'd to the per-VM dnsproxy. VMs cannot query external resolvers directly.
- **DoT** (DNS-over-TLS): TCP port 853 is blocked.
- **DoH** (DNS-over-HTTPS): handled by SNI filtering on port 443. Connections to known resolver hostnames are denied in `custom` mode.
- **DoQ** (DNS-over-QUIC): blocked by the UDP default-deny.
- **ECH** SvcParam records are stripped from DNS responses to prevent Encrypted ClientHello from hiding the SNI.

### Cross-VM isolation

Traffic to internal subnets is blocked to prevent VMs from reaching each other:

| Subnet | Purpose |
| --- | --- |
| `198.19.0.0/16` | VM link subnet (guest/host IPs) |
| `172.16.0.0/16` | Legacy VM address block |
| `10.200.0.0/16` | Veth transit subnet |
| `10.201.0.0/16` | Mesh subnet |

## Reserved ports

The following port ranges are used by DNS and SNI proxy functionality and cannot be used as published ports:

| Range | Purpose |
| --- | --- |
| `10053-10307` | DNS proxy (one port per VM slot) |
| `10443-10697` | SNI proxy (one port per VM slot) |
| `10698-10952` | HTTP proxy (one port per VM slot) |

## Troubleshooting

### Check nftables support

`vmsan doctor` detects nftables kernel support and host firewalls. Run it to verify your system is properly configured:

```bash
vmsan doctor
```

### Host firewall interference

If the host runs ufw or firewalld, their FORWARD chains may interfere with vmsan traffic. In nftables, multiple tables each have independent chains at the same hook -- if any table's chain drops a packet, it's dropped.

To fix this, add explicit allow rules for vmsan traffic in your host firewall. For example, with ufw:

```bash
sudo ufw allow in on veth-h-+
sudo ufw route allow in on veth-h-+
```

### DNS not resolving

If DNS queries from a VM fail or time out:

1. Check that vmsan-gateway is running: `ps aux | grep vmsan-gateway`
2. Verify the dnsproxy sidecar for the VM slot is active: `vmsan logs dns <vm-id>`
3. Ensure the VM's DNS is DNAT'd to the host proxy: `sudo nft list table ip vmsan_<vmId>`

### TLS connections timing out

If HTTPS requests hang or fail with connection reset:

1. Verify the tcpproxy is running for the VM slot.
2. In `custom` mode, check that the target domain is in `--allowed-domain`. SNI-based filtering will deny connections to unlisted domains.
3. Check if the server uses TLS without SNI (IP-only TLS). This is denied by default in `custom` mode.

### Inspecting active rules

To inspect the nftables rules for a specific VM:

```bash
sudo nft list table ip vmsan_<vmId>
```

To list all vmsan-managed tables:

```bash
sudo nft list tables | grep vmsan
```
