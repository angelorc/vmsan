vmsan.toml is a declarative configuration file that defines your project's services, accessories, and deployment settings. It is the single source of truth for how your application runs on vmsan.

## Getting started

Use `vmsan init` to generate a vmsan.toml for your project. The wizard auto-detects your runtime, build command, and start command from project files.

```bash
vmsan init
```

After generating the file, validate it before deploying:

```bash
vmsan validate
```

::tip
`vmsan validate` checks for required fields, valid runtimes, valid network policies, valid accessory types, circular dependencies, and even suggests fixes for typos using Levenshtein distance matching.
::

## Single-service format

For simple projects with a single service, use the flat format:

```toml
project = "my-app"
runtime = "node24"
build = "npm run build"
start = "npm start"
```

## Multi-service format

For projects with multiple services and accessories, use the structured format:

```toml
[services.web]
runtime = "node24"
build = "npm run build"
start = "npm start"
memory = 512
vcpus = 2
disk = "20gb"
publish_ports = [8080]
network_policy = "allow-all"
depends_on = ["db"]
connect_to = ["api:3000"]
service = "web"

[services.api]
runtime = "node24"
build = "npm run build"
start = "node server.js"
connect_to = ["db:5432"]

[accessories.db]
type = "postgres"
version = "16"
env = { POSTGRES_PASSWORD = "${{ secrets.DB_PASSWORD }}" }

[accessories.cache]
type = "redis"
version = "7"

[deploy]
release = "production"

[tunnel]
hostname = "example.com"
```

## Service options

| Field | Type | Default | Description |
| --- | --- | --- | --- |
| `runtime` | string | `base` | Runtime label (`base`, `node22`, `node24`, `python3.13`, `go`) |
| `build` | string | none | Build command to run during deployment |
| `start` | string | none | Start command to run the service |
| `env` | table | none | Environment variables (supports secret references) |
| `depends_on` | array | none | Services that must be deployed before this one |
| `connect_to` | array | none | Mesh connections (`service:port` format) |
| `service` | string | none | Register this VM as a mesh service name |
| `publish_ports` | array | none | Ports to expose on the host |
| `memory` | int | `128` | Memory in MiB |
| `vcpus` | int | `1` | Number of vCPUs |
| `disk` | string | `10gb` | Root disk size |
| `network_policy` | string | `allow-all` | Network policy (`allow-all`, `custom`, `deny-all`) |
| `allowed_domains` | array | none | Allowed domains for `custom` network policy |
| `health_check` | table | none | Health check configuration |

## Accessories

Accessories are built-in services with pre-built rootfs images that are auto-configured for your project.

| Type | Versions | Description |
| --- | --- | --- |
| `postgres` | 16 | PostgreSQL database with auto-configured credentials |
| `redis` | 7 | Redis in-memory store, ready to use |
| `mysql` | — | MySQL database |

Each accessory supports these fields:

| Field | Required | Description |
| --- | --- | --- |
| `type` | yes | Accessory type (`postgres`, `redis`, `mysql`) |
| `version` | no | Version to use |
| `env` | no | Environment variables to pass to the accessory |

::note
Accessories are deployed before services in the dependency graph. Credentials and connection strings are auto-injected into dependent services.
::

## Health checks

Configure health checks to verify your service is running correctly:

```toml
[services.web.health_check]
type = "http"
path = "/health"
port = 8080
interval = 10
timeout = 5
retries = 3
```

### Health check types

| Type | Description |
| --- | --- |
| `http` | Send an HTTP GET request to `path` on `port`. Expects a 2xx response. |
| `tcp` | Attempt a TCP connection to `port`. Succeeds if the connection is established. |
| `command` | Run a command inside the VM. Succeeds if exit code is 0. |

## Secret references

Use the `${{ secrets.KEY }}` syntax to reference encrypted secrets in environment variables:

```toml
[services.api]
runtime = "node24"
env = { DATABASE_URL = "${{ secrets.DATABASE_URL }}" }

[accessories.db]
type = "postgres"
env = { POSTGRES_PASSWORD = "${{ secrets.DB_PASSWORD }}" }
```

Secrets are managed with `vmsan secrets` and injected at deploy time. They are never stored in vmsan.toml. See the [Secrets](/guide/secrets) guide for details.

## Auto-detection

`vmsan init` detects your project's runtime and commands from these files:

| File | Detected runtime |
| --- | --- |
| `package.json` | Node.js (node22 or node24) |
| `go.mod` | Go |
| `requirements.txt` / `pyproject.toml` | Python |
| `Cargo.toml` | Rust |
| `Dockerfile` | Docker-based build |

## Validation

Run `vmsan validate` to check your configuration before deploying:

```bash
vmsan validate
```

The validator checks for:

- Required fields (`runtime`, `start`)
- Valid runtime labels
- Valid network policy values
- Valid accessory types
- Circular dependencies in `depends_on`
- Typo suggestions using Levenshtein distance (e.g., "nodee24" suggests "node24")

::warning
`vmsan up` also runs validation before deploying. If validation fails, no services are deployed.
::
