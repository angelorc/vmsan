vmsan uses a multi-binary architecture that separates privileges, isolates concerns, and scales from a single machine to a fleet of hosts.

## Overview

```
                    ┌──────────────────────────────────┐
                    │         vmsan-server             │
                    │   (control plane, SQLite, HTTP)  │
                    └──────────┬───────────────────────┘
                               │ HTTP (pull-based sync)
                    ┌──────────▼───────────────────────┐
                    │       vmsan-agent-host           │
                    │   (worker daemon, heartbeats)    │
                    └──────────┬───────────────────────┘
                               │
  ┌─────────┐     Unix socket  │    ┌──────────────────┐
  │  vmsan  │ ─── gRPC ──────► │ ── │  vmsan-gateway   │
  │  (CLI)  │                  │    │  (privileged)    │
  └─────────┘                  │    └────────┬─────────┘
                               │             │
                               │    ┌────────▼──────────┐
                               │    │   Firecracker     │
                               │    │   (microVM)       │
                               │    │  ┌──────────────┐ │
                               │    │  │ vmsan-agent  │ │
                               │    │  │ (port 9119)  │ │
                               │    │  └──────────────┘ │
                               │    └───────────────────┘
```

## vmsan (CLI)

The CLI is an unprivileged Go binary built with [Cobra](https://github.com/spf13/cobra). It is the primary user interface for all vmsan operations.

**Responsibilities:**
- Parsing `vmsan.toml` project configuration
- Sending gRPC requests to the gateway over the Unix socket
- Formatting output (tables, JSON, streaming logs)
- Managing project-level workflows (`init`, `up`, `down`, `deploy`, `status`)

The CLI never touches Firecracker, networking, or filesystem operations directly. All privileged work is delegated to the gateway.

```bash
# All of these run without sudo
vmsan create --runtime node22
vmsan up
vmsan logs my-service --follow
vmsan connect vm-a3f7b2c
```

## vmsan-gateway

The gateway is a privileged daemon that runs as a systemd service under root. It owns all operations that require elevated permissions.

**Responsibilities:**
- Managing VM lifecycle (create, start, stop, remove)
- Setting up network namespaces, veth pairs, and TAP devices
- Configuring nftables firewall rules (per-VM tables)
- Running Firecracker via the jailer (chroot, cgroups, seccomp)
- Managing per-VM DNS proxy and SNI proxy for egress filtering
- Allocating IP addresses and network slots

**Socket:** `/run/vmsan/gateway.sock`
**Protocol:** gRPC over Unix socket (Protobuf)

::note
The gateway creates one nftables table per VM (`vmsan_<vmId>`) with FORWARD, prerouting, and postrouting chains. Rules are applied atomically using the `google/nftables` netlink library.
::

## vmsan-server

The server is the control plane for multi-host deployments. It runs as a standalone HTTP server with a SQLite backend.

**Responsibilities:**
- Host registration via join tokens
- VM scheduling and placement decisions
- Sync engine for distributing commands to workers
- Fleet-wide status aggregation
- State persistence in SQLite

The server does not manage VMs directly. It coordinates with `vmsan-agent-host` workers that execute operations on their local gateways.

::tip
For single-host usage, you do not need vmsan-server or vmsan-agent-host. The CLI communicates directly with the local gateway.
::

## vmsan-agent-host

The agent-host is a worker daemon that runs on each remote server in a multi-host deployment. It joins the control plane and executes VM operations locally.

**Responsibilities:**
- Registering with the control plane using a join token
- Sending heartbeats every 30 seconds
- Pulling commands from the sync engine every 10 seconds
- Executing VM operations via the local gateway
- Reporting host metrics and VM state back to the server

**Communication:** HTTP REST with pull-based sync (agent-host polls the server)

## vmsan-agent (in-VM)

The in-VM agent is a Go HTTP server that runs inside every Firecracker microVM on port 9119. It is the bridge between the host and the guest operating system.

**Responsibilities:**
- Interactive shell access via WebSocket PTY
- File upload and download operations
- Command execution with stdout/stderr streaming
- Health check endpoint for liveness probes
- Log forwarding to the host

::warning
The in-VM agent is not the same as vmsan-agent-host. The agent runs *inside* the VM guest, while agent-host runs on the *host* machine as a control plane worker.
::

## Communication protocols

| Path | Protocol | Transport |
| --- | --- | --- |
| CLI to Gateway | gRPC / Protobuf | Unix socket (`/run/vmsan/gateway.sock`) |
| Server to Agent-Host | HTTP REST | TCP (pull-based sync, agent-host polls) |
| CLI to in-VM Agent | HTTP / WebSocket | TCP to VM IP on port 9119 |
| Gateway to Firecracker | Firecracker API | Unix socket (per-VM, inside jailer chroot) |

## Privilege separation

vmsan follows the same trust model as Docker: the CLI runs as your regular user, while a privileged daemon handles operations that require root.

```
  Unprivileged                    Privileged (root)
  ─────────────                   ─────────────────
  vmsan (CLI)  ── Unix socket ──► vmsan-gateway
  vmsan-server                    vmsan-agent-host
```

This design means:
- **No sudo required** for day-to-day CLI operations
- **Socket permissions** are managed via the `vmsan` group --- users in this group can communicate with the gateway
- **Blast radius is limited** --- the CLI cannot directly modify networking, filesystems, or firewall rules
- **Auditability** --- all privileged operations flow through a single daemon with structured logging

::caution
The gateway runs as root because it must create network namespaces, configure nftables rules, and invoke the Firecracker jailer. Do not run the gateway as an unprivileged user.
::
