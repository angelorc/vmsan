vmsan is a Firecracker microVM platform that lets you create, deploy, and manage isolated microVMs with a privilege-separated architecture, declarative configuration, and multi-host scaling.

## What is vmsan?

vmsan wraps [Firecracker](https://github.com/firecracker-microvm/firecracker) to provide a complete platform for running lightweight, security-isolated virtual machines. It uses a multi-binary architecture where an unprivileged CLI communicates with a privileged gateway daemon over a Unix socket --- the same trust model as Docker.

Each VM runs inside a jailer with its own network namespace, seccomp filters, and cgroup resource limits. A Go-based agent inside the VM exposes an HTTP API for shell access, file operations, and command execution.

### Architecture at a glance

vmsan is built from four binaries that separate concerns cleanly:

| Binary | Role | Privilege |
| --- | --- | --- |
| `vmsan` | CLI for user interaction | Unprivileged |
| `vmsan-gateway` | VM lifecycle, networking, firewalls | Root (systemd service) |
| `vmsan-server` | Multi-host control plane | Unprivileged |
| `vmsan-agent-host` | Worker daemon on remote hosts | Root |

The CLI sends gRPC calls to the gateway over a Unix socket. For multi-host deployments, `vmsan-server` coordinates across hosts via HTTP, and `vmsan-agent-host` workers pull commands from the control plane.

See the [Architecture](/getting-started/architecture) page for a detailed breakdown of each component and how they communicate.

## What you can do

- **Manage the full VM lifecycle** --- Create, start, stop, and remove microVMs with simple commands
- **Deploy declaratively** --- Define services, resources, and secrets in `vmsan.toml` and bring everything up with `vmsan up`
- **Isolate network access** --- Use allow-all, deny-all, or custom policies with domain and CIDR allowlists and defense-in-depth egress filtering
- **Connect interactively** --- Open a shell to any running VM via WebSocket PTY
- **Stream logs** --- Follow VM output in real time with `vmsan logs --follow`
- **Mesh networking** --- VMs discover each other by name with automatic DNS and routing
- **Manage secrets** --- Inject secrets into VMs securely without baking them into images
- **Transfer files** --- Upload and download files without SSH
- **Use Docker images** --- Build root filesystems from any Docker or OCI image with `--from-image`
- **Choose your runtime** --- Pick from `base`, `node22`, `node24`, or `python3.13`
- **Snapshot and restore** --- Capture VM state and restore it later
- **Scale across hosts** --- Register multiple servers with a control plane and schedule VMs across your fleet
- **Script everything** --- Get structured JSON output with the `--json` flag

## When to use vmsan

Use vmsan when you need to:

- Run untrusted code in an isolated sandbox
- Create disposable development environments
- Deploy multi-service applications with declarative config
- Test software across different runtimes
- Provide isolated compute environments for CI/CD or multi-tenant workloads
- Build your own PaaS on bare-metal or cloud servers
- Run workloads across multiple hosts with centralized management
